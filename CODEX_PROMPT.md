```
Project context:

  FixedExpensesEditor (dynamic list of fixed expenses, used in Onboarding and Settings).

- README.md → general project documentation.

Your task:
Maintain or extend the Python (FastAPI) backend in the repo root's main.py, and its pyproject.toml, so that it
keeps satisfying EXACTLY the contract frontend/lib/api.ts expects. Do not modify the frontend unless strictly
necessary to make the integration work.

Mandatory technical requirements:
1. Framework: FastAPI + Uvicorn + Pydantic + python-dotenv + openai (SDK >=1.50). Package manager: uv (use
   "uv add fastapi 'uvicorn[standard]' pydantic python-dotenv openai" and "uv run uvicorn main:app --reload --port 8000").
2. CORS: must allow origin "http://localhost:3000" (where the frontend runs), all methods and headers.
3. The server must run on port 8000 (0.0.0.0:8000).
4. In-memory state (Python lists/dicts), no database required.
5. Branding: the product name is "confIA" and the tagline is "Yo confío, tú confIA". Use "confIA" (not "Luminous"

6. IMPORTANT: the project name in pyproject.toml must be "confia-backend" (NOT "openai"), to avoid confusion with
   the real PyPI package "openai" the project itself depends on.

Exact endpoints to implement (routes, methods, and JSON response shape):

- GET /api/health
  Response: { status: string, time: string (ISO 8601) }

- POST /api/onboarding
  Body: { ingreso: float, gastos_fijos: FixedExpense[], ahorro_pct: float }
  Response: { ok: bool, trust_score: int }  // trust_score between 0 and 20, proportional to ahorro_pct
  NOTE: the profile no longer has standalone "capital" or "gastos" fields (removed in the redesign). Recurring
  expenses live as a list of FixedExpense = { name: string, day?: string|null, varies: bool, amount?: number|null }.
  "varies: true" means the amount changes every month (e.g. an electricity bill), so "amount" may be null.

- GET /api/profile
  Response: { ingreso: float, gastos_fijos: FixedExpense[], ahorro_pct: float }

- GET /api/dashboard
  Response: {
    trust_score: { value: int, max: int, label: string },   // label computed from thresholds (Excellent/Healthy/Fair/Needs attention)
    balance: { ingresos: float, gastos: float, meta_ahorro_pct: float },
    ai_insight: string,
    recent: Transaction[],                                   // last 3 transactions
    missing_data: { id: int, title: string }[],              // transactions with completed=false (missing receipt/evidence)
    fixed_expenses: (FixedExpense & { status: "pagado"|"pendiente"|"falta_comprobante", icon: string })[],
      // each fixed expense in the profile is matched (case-insensitive, substring) against transaction titles:
      // no match -> "pendiente"; match with completed=false -> "falta_comprobante"; match with completed=true -> "pagado"
    safety_cushion_days: int,
    daily_budget: { spent: float, remaining: float, limit_pct: int }
  }

- GET /api/transactions?category=&kind=
  kind can be "income" or "expense" (optional), category is free text (optional, case-insensitive)
  Response: { transactions: Transaction[], total: int }

- POST /api/transactions
  Body: { title: string, category: string, amount: float, icon?: string }
  Response: Transaction (the created transaction, with a new auto-incrementing id, date="Today", completed=true by default)

- PATCH /api/transactions/{id}
  Body: Partial<{ title, category, amount, date, icon, completed }>  // all fields optional, only present ones are updated
  Response: the updated Transaction. If the id doesn't exist, return 404.

- DELETE /api/transactions/{id}
  Response: { ok: bool }. If the id doesn't exist, return 404.

- GET /api/analytics
  Response: {
    tranquility_trend: number[],           // series of points for a chart
    score_today: int,
    spend_pace: { today: number, yesterday_same_day: number },
    top_categories: { name: string, pct: number, amount: number, color: string }[],  // grouped and sorted from real transactions
    income_vs_expense: { ingresos: number, gastos: number },
    small_treats: { amount: number, equivalent: string },
    safety_cushion_days: int,
    daily_budget: { spent: number, remaining: number, limit_pct: number },
    ai_advice: string
  }

- POST /api/ai-chat
  Body: { message: string }
  Response: { reply: string, detected_category: string, detected_amount: number, detected_type: "income"|"expense" }
  Logic: parse the message text looking for a numeric amount, a known category
  (Transporte, Alimentación, Ocio, Servicios, Salario) and income keywords (ingreso, depósito, nómina, sueldo, etc.)
  to decide detected_type. If an amount is detected, reply confirming the detected expense/income; otherwise give
  generic advice signed as confIA. This simulates AI — it does NOT require a real external API.

- POST /api/receipt-scan
  Body: { image: string }  // base64 data URL, e.g. "data:image/png;base64,...."
  Response: { title: string, category: string, amount: number, type: "income"|"expense" }
  Logic: use the official OpenAI SDK (model "gpt-4o-mini", image + text content, response_format json_object) to
  read a real receipt and extract its data. category must be exactly one of:
  Transporte, Alimentación, Ocio, Servicios, Salario, Otros. If no OPENAI_API_KEY is configured (environment
  variable, loaded via python-dotenv from a .env file), respond with 503 and a clear message instead of failing
  silently — the frontend must keep working in manual mode without this key.

Where:
  Transaction = { id: int, title: string, category: string, amount: number, date: string, icon: string, completed: bool }
  (negative amount = expense, positive = income; icon is a Material Symbols icon name, e.g. "local_taxi",
  "shopping_basket", "payments", "restaurant", "subscriptions", "receipt_long"; completed=false means the receipt/
  evidence is missing and feeds the Home alerts)

Seed the backend with 5-8 varied sample transactions (income and expenses, different categories) so the demo
looks complete from the very first second, without the user having to create data manually.

Mandatory final verification (do not consider the task done without this):

Do not invent endpoints or change field names: they must match EXACTLY what
```
